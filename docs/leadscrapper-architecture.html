1. System Overview & High-Level Architecture

LeadScrapper is a lead generation platform that scrapes Google Maps for business listings using Apify, scores them by lead quality, manages them in a CRM-like interface, extracts emails, generates outreach emails, and can create full websites for leads via Map2Web integration.

2. The Complete Lead Search Flow (10 Steps)

1 User Input

User enters a keyword (e.g., "plumber"), selects country/state/city or ZIP code, and clicks Search. The LeadSearch component collects: keyword, locationQuery (or postalCode + countryCode), maxCrawledPlacesPerSearch.

File: components/LeadSearch.jsx

2 Credit Pre-Check

Before calling Apify, the proxy checks the user's credit balance via credits_get_balance(). This calls Makerkit's /api/supabase/credits/get endpoint. If balance < 0.01 credit, returns 402 (Insufficient Credits).

File: apify-proxy.php (lines 428-455)

3 Supabase Cache Check

Builds a cache key from keyword|location (e.g., "plumber|zip:90001|us"). Queries the leadscrapper_leads_data table for existing rows with UserEmail='__cache__'. If cache has any places for this key, NEVER re-scrapes — serves from cache directly with a synthetic runId.

File: apify-proxy.php (lines 457-508), lib/supabase_cache.php

4 Apify Scraping (on cache miss)

On cache miss, calls the Apify compass~crawler-google-places actor. The proxy rotates through up to 21 API keys (APIFY_KEY_1 through APIFY_KEY_21), advancing to the next key per query. If a key returns 402/403/429, auto-rotates to the next.

File: apify-proxy.php (lines 511-549)

5 Poll & Dataset Fetch

Frontend polls ?action=check&runId=... every 3 seconds until SUCCEEDED. Then fetches the dataset via ?action=dataset&datasetId=.... The proxy fetches the FULL Apify dataset (up to 5000 items) to merge into the shared cache.

File: components/LeadSearch.jsx (polling logic), apify-proxy.php (lines 592-706)

6 Cache Merge

After fetching live Apify data, $supabaseCache->mergePlaces() deduplicates by placeId and upserts new places into leadscrapper_leads_data with UserEmail='__cache__'. Future identical queries are served from this cache for free.

File: lib/supabase_cache.php (lines 82-108)

7 Credit Slicing

applyCreditSlice() computes what the user receives:

  • Fetches user's already-delivered placeIds (free, already paid)
  • Checks extras queue (overflow from previous searches)
  • Delivers extras FIRST, then new pool leads, capped at 100 per search
  • Charges leadCount * 0.01 credits via Makerkit's /api/supabase/credits/deduct-leads
  • Records delivered IDs, enqueues overflow, logs the search

File: lib/credits.php (lines 514-592), apify-proxy.php (lines 186-310)

8 Lead Scoring

Frontend scores each lead 0-10 based on: unclaimed GBP (+25), no website (+20), no SSL (+10), no reviews (+20), low rating (+10), no social (+5). Leads are tagged as "Hot Lead" (7.5+), "Warm Lead" (5+), "Cool Lead" (2.5+), or "Low Priority".

File: components/LeadSearch.jsx (lines 40-77), utils/helpers.js

9 Email Extraction (Background)

For leads with websites, the frontend calls extract-email.php?url=... which scrapes the business homepage (then /contact, /about pages) for email addresses. Results are cached in localStorage for 24 hours.

File: extract-email.php, utils/emailExtractor.js

10 Save to Supabase

User clicks Save (or auto-bulk-saves). leads-proxy.php upserts per-user state rows into user_leadscrapper_leads with Status='delivered'. Business data stays in the pool table; only user-editable metadata (Notes, LeadScore, ManagerStatus) is stored per-user.

File: leads-proxy.php (lines 180-258)

5. Apify Integration & Key Rotation

Actor Used

compass~crawler-google-places — a free/public Apify actor for scraping Google Maps business listings.

Key Rotation System

FeatureDetail
Max KeysUp to 21 keys (APIFY_KEY_1 through APIFY_KEY_21)
Rotation TriggerHTTP 402 (quota), 403 (disabled), 429 (rate limit)
Per-Query RotationEach new search advances to next key (round-robin)
Run BindingrunKeyMap maps runId → keyIndex so check/dataset use the same key
State Persistence.apify_key_state.json file (currentIndex + runKeyMap + expiry)
ExpiryrunKeyMap entries pruned after 24 hours

Request Flow

Action: run
  → Credit pre-check (Makerkit balance)
  → Cache check (Supabase)
  → If cache HIT: return synthetic runId immediately
  → If cache MISS: POST to Apify actor with current key
  → On success: map runId → keyIndex, advance to next key

Action: check
  → If cached runId: return SUCCEEDED immediately
  → Else: GET actor-runs/{runId} with same key that started it

Action: dataset
  → If cached runId: decode cacheKey from runId, fetch from Supabase cache
  → Apply credit slice, return sliced results
  → If live: GET dataset items, merge into cache, apply credit slice

Extras Queue Logic

When a search returns more leads than the user's balance allows, the overflow is queued as Status='queued' in user_leadscrapper_leads. On the next search for the same cacheKey, these extras are delivered FIRST (free — already counted). Only truly new leads are charged.

9. Email Extraction System

Flow

  1. Frontend calls extract-email.php?url=https://business-website.com
  2. PHP fetches the homepage HTML via cURL
  3. Extracts emails from mailto: links and general regex matching
  4. Filters blacklisted domains (wixpress.com, sentry.io, etc.) and non-person emails (noreply, postmaster)
  5. If no emails found on homepage, tries /contact, /contact-us, /about, /about-us
  6. Returns { success, emails: [], source }

Client-Side Caching

utils/emailExtractor.js caches results in localStorage with a 24-hour TTL. Keyed by domain. Even empty results are cached to avoid re-scraping.

Blacklisted Domains

wixpress.com, sentry.io, w3.org, schema.org, googleapis.com, google.com, facebook.com, twitter.com, example.com, wordpress.org, jquery.com, cloudflare.com, gstatic.com, gravatar.com, wp.com, squarespace.com, wix.com, weebly.com, godaddy.com

10. Map2Web Site Generation Integration

LeadScrapper can generate Basic/Gold/Premium websites for any lead via Map2Web.

Flow

  1. User clicks "Generate Sites" on one or more leads in LeadManager
  2. app.jsx calls generateSiteForLead(lead)
  3. utils/map2web.js converts the lead to a Map2Web business object via leadToBusiness()
  4. For each tier (1=Basic, 2=Gold, 3=Premium):
    • POST /map2web-proxy.php?path=build → returns { filePath, html, repoName }
    • POST /map2web-proxy.php?path=publish → pushes to GitHub Pages
  5. POST /map2web-proxy.php?path=log → records in Map2Web history
  6. Tier URLs are persisted back to Supabase (Tier1, Tier2, Tier3 columns)

11. Authentication & Security

Three-Stage Gate

  1. Cloudflare Turnstile — CAPTCHA gate shown first (skipped on localhost and embed mode)
  2. OTP Login — email-based 6-digit OTP sent via Resend API, verified client-side
  3. Inactivity Timer — auto-logout after 10 minutes of no mouse/keyboard activity

Embed Mode (Makerkit Integration)

When ?embed=true, the app skips Turnstile + OTP. The email is passed via URL param from the parent Makerkit page. This assumes Makerkit has already authenticated the user.

Security Considerations

1. Client-Side OTP Verification OTP is generated and verified in the browser. The OTP value is visible in JavaScript. Same trade-off as Map2Web.
2. Email in URL Param In embed mode, the user email is passed as ?email=... in the URL. This is visible in browser history, server logs, and referrer headers.
3. API Keys Hidden All Apify API keys stay server-side in .env. The PHP proxy prevents key exposure to the browser.

14. Architecture Observations & Gotchas

1. PHP Files are Standalone Proxies No framework, no router, no middleware. Each .php file is its own endpoint. The build script copies them to dist/ alongside the bundled React app.
2. Two Supabase Instances LeadScrapper's Supabase holds leads data + local credit audit tables. Makerkit's Supabase (same or different instance) holds the authoritative user_credits table. The PHP layer bridges between them.
3. Cache is Permanent Once a cacheKey has ANY entries in leadscrapper_leads_data, Apify is NEVER called again for that query. The TTL gate was removed. This saves API costs but means stale data persists.
4. Three Lead States The same lead can exist in three states simultaneously: as a pool row (shared data), as a delivered row (user received it), and as a queued row (overflow from previous search). The system deduplicates carefully.
5. Client-Side OTP + Email in URL Same security trade-offs as Map2Web. OTP verified in browser, email visible in URL params during embed mode.
6. Apify Key Rotation is File-Based .apify_key_state.json persists the current key index. On multi-instance deployments, this could cause race conditions. Works fine for single-server setup.
7. Embed Mode Bypasses All Auth When ?embed=true, the app trusts the email URL param completely. Security depends on the parent Makerkit app properly authenticating users before embedding.
8. Map2Web is a Reverse Proxy The LeadScrapper server proxies all Map2Web API calls through map2web-proxy.php. This avoids CORS issues and keeps the service token server-side. Whitelist: scrape, deduct, build, publish, log.
9. Supabase Cache Replaces Firebase The cache was migrated from Firebase Firestore to Supabase (leadscrapper_leads_data table). The SupabaseCache class is a drop-in replacement with the same interface.
10. Makerkit is the Source of Truth for Credits LeadScrapper never modifies the user_credits table directly. All balance reads and deductions go through Makerkit's Next.js API routes, which use the Supabase admin client.
